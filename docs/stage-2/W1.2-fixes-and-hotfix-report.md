## 🔴 Stage 2 Hotfix — empty line financials (your point #2, "خطأ فادح")

**Classified as a Data-Integrity / Financial display bug and fixed immediately, per the Stage 2 Hotfix rule.**

**Root cause (investigated against the live API):** the claim-line DTO uses **`medicalServiceName`** — my code (and the original) read `line.serviceName`, so the service name was blank. And the line DTO has **no `companyShare`/`patientShare` fields at all** — only `coveragePercent` (75), `patientSharePercent` (25), `totalPrice`, `refusedAmount`. Reading the non-existent fields produced **0.00**. This was a **pre-existing bug** that my W1.2 columns exposed.

**Fix (honest, no financial misrepresentation):**
- **Service name** → now reads `medicalServiceName` (falls back to `medicalServiceCode`). The column is populated again.
- **Company share / Member co-pay** → derived from the line's **own coverage percentages** applied to the accepted amount:
  - `member co-pay = (totalPrice − refused) × patientSharePercent%`
  - `company share = (totalPrice − refused) × coveragePercent%`
- **Verified against authoritative backend figures** (claim 251): my **member co-pay 187.50 exactly equals the backend's `patientCoPay` (187.50)** ✅. Company share 562.50 is the benefit-covered portion (75% of 750); the claim's **approved 506.25** = that minus the **10% provider discount** (`providerDiscountPercent: 10`) — a claim-level TPA↔provider arrangement, correctly shown separately at claim level.
- **Rejected / benefit cap / remaining** now render correctly (0.00 / — reflect the real data — this claim has nothing rejected and no benefit cap set).

**Transparency note:** the per-line **company share (562.50)** is the *benefit-covered* amount (before the provider discount); the claim's *approved* (506.25) is the *net paid to the provider after* that discount. Both are correct and now shown honestly. If you want the per-line figure to instead reflect the post-discount net, that requires the backend to return a per-line discounted breakdown — a small backend enhancement I did **not** guess at (financial single-source-of-truth stays on the server).
