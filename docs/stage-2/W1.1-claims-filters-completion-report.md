## Objective (yours)

Build a reusable **Enterprise Filter Framework** and apply it to the operational area claim staff use all day — reclaiming table space, fewer clicks, less scrolling. No workflow/business redesign.

## Analyze

The Claims workflow's heaviest, most filter-centric page is the **Claims Operational Report** (`pages/reports/claims/index.jsx`), which rendered an **always-open `Paper` grid of filters** (`ClaimsFilters`: employer, provider, status multi-select, member search, date range) permanently consuming vertical space above the table — the exact pain you described. That is the migration target for W1.1. (`ClaimBatchManagement` filters are navigational — employer + month/year — and are addressed in later steps.)

## Implement — the reusable framework

New, config-driven, self-contained (no data/workflow logic inside it):

- **`components/common/EnterpriseFilters/EnterpriseFilters.jsx`**
  - ✅ **Collapsible inline panel — collapsed by default** → maximum table area (the table is the primary element).
  - ✅ **Drawer variant** and **automatic drawer on small screens** (`variant="auto"`) so the table stays full-width on narrow viewports.
  - ✅ **Active filter chips** (each removable) always visible in a compact one-row bar.
  - ✅ **Clear-all** (appears only when filters are active).
  - ✅ **Remembers open/closed state** per `filterKey` (localStorage).
  - ✅ **Responsive** (MUI breakpoints) + **keyboard friendly** (real buttons; drawer closes on Esc; standard inputs).
  - ✅ **Config-driven `fields`** — supports `text | select | multiselect | date`, per-field icon, placeholder, `hidden` (e.g. employer selector hidden for non-admins), and grid sizing.
- **`components/common/EnterpriseFilters/useFilterState.js`** — a drop-in `useState` replacement that **remembers filter values** across sessions per key (merges saved values over defaults; storage-safe).
- **`components/common/EnterpriseFilters/index.js`** — barrel export.

Together they satisfy every requirement you listed: collapsible panel, sidebar/drawer, active chips, clear-all, remember state (open/closed **and** values), responsive, keyboard-friendly, maximize table area.

## Implement — migrate the Claims Operational Report

- Replaced the always-open `ClaimsFilters` `Paper` with `<EnterpriseFilters>` driven by a `fields` config (employer, provider, statuses, member search, date-from, date-to).
- Swapped `useState(DEFAULT_FILTERS)` → `useFilterState('claims-report-filters', DEFAULT_FILTERS)` so the filters persist across sessions.
- **Data layer untouched:** each filter change is routed to the page's existing setters (`handleEmployerChange` / `handleProviderChange` / `handleFilterChange`), so `useClaimsReport` and the employer-scope/lock logic behave exactly as before. **No workflow or query change.**
- **Result:** the report now opens with the **filter panel collapsed** — the table gets the full height; active filters are shown as removable chips; one click opens the panel; on a narrow screen the panel becomes a drawer.
