# 13 — Printing Review

**Score: 60/100 (C−)** · Reviewed against the Reporting & Printing Constitution's Printing section and the UI/UX Constitution's Printing principles.

---

## Print Preview

`react-to-print` is a confirmed dependency, and `pages/claims/ClaimStatementPreview.jsx` exists as a dedicated preview surface — the fundamental capability is present. Per-page consistency (does every printable document offer the same preview experience) was not exhaustively verified across all report/document types in this pass.

## Professional Layout / A4

The backend PDF pipeline (OpenPDF + Flying Saucer, see `14-pdf-review.md`) targets standard document rendering; A4-specific page-size configuration was not independently confirmed at the CSS/print-stylesheet level for the browser-native print path (`react-to-print`) as distinct from the server-rendered PDF path — these are two different rendering pipelines (client CSS-print vs. server PDF-render) and **should produce visually identical output** per the UI/UX Constitution's explicit requirement ("*Printed output must match PDF output*"). This equivalence was not verified in this assessment and is a concrete, checkable follow-up item.

## Arabic / RTL

Inherits the platform's strong, centralized RTL architecture (`RTLLayout.jsx`, `stylis-plugin-rtl`) for anything rendered through the normal React component tree. The server-side PDF path uses a separate rendering engine (Thymeleaf → Flying Saucer) that does **not** go through the frontend's RTL infrastructure at all — meaning RTL correctness for printed/PDF documents depends entirely on the Thymeleaf template's own explicit `dir="rtl"`/CSS handling, a second, independent place RTL must be gotten right. This is architecturally reasonable (client and server rendering are necessarily separate systems) but means RTL quality assurance must happen twice, not once.

## Margins / Headers / Footers

Not independently verified against the Constitution's explicit list (Company Logo, Report Title, Report Number, Generated By, Generated Date/Time, Page Number, Applied Filters, Footer, Confidentiality Label) for either the print-preview path or the PDF path. This is the same gap flagged in `12-reporting-review.md` and should be resolved by the same standardized-header-component recommendation.

## Pagination

Automatic page breaks and repeated table headers across pages are explicit Constitution requirements ("*Every printable report should support: Automatic Page Breaks, Repeated Table Headers*"). Flying Saucer (the PDF rendering engine in use) does support CSS-based page-break and repeated-header directives, but whether the actual Thymeleaf templates use them was not verified in this pass — recommend a targeted check on any report likely to span multiple pages (Beneficiaries Report, Financial Consolidation).

## Watermarks / Signature / Stamp Areas

The Constitution explicitly requires configurable watermark support (Draft/Copy/Confidential/Cancelled/Paid/Approved/Rejected) and signature/stamp areas for official documents. **No evidence of watermark or signature/stamp-area support was found** in this assessment — this appears to be a genuine capability gap rather than an implementation-quality issue, since no partial/broken implementation was found either. This is the most concrete, actionable printing gap identified.

## Overall Printing Maturity

The fundamentals (A4-target PDF generation, Arabic font embedding, print-preview capability) are present and represent real, correct technical choices. What's missing is the **enterprise document-identity layer** the Constitution explicitly asks for (watermarks, signature/stamp areas, document numbers, confidentiality labels) — features that matter for a document's legitimacy as an official business/legal artifact (per the Constitution's Final Principle: "*every printed page should be suitable for executive review, provider communication and regulatory submission without manual editing*"). Today, a printed claim statement or settlement report likely lacks the visual markers that would let a provider or regulator immediately recognize it as an authoritative, current-status Waad TPA document versus, say, a draft or an outdated reprint.

---

## Findings Requiring Action

1. **(High)** Verify browser-print (`react-to-print`) output visually matches server-rendered PDF output, per the Constitution's explicit equivalence requirement — if they diverge, standardize on one canonical rendering path for anything that must be print-identical to its PDF.
2. **(Medium)** Add configurable watermark support (at minimum Draft/Approved/Rejected/Paid, matching claim/payment status vocabulary already in the domain model).
3. **(Medium)** Add signature/stamp area placeholders to official documents (claim statements, settlement reports) — configurable, not hardcoded, per the Constitution's Digital Signature section ("*do not hardcode signature images... signature areas must remain configurable*").
4. **(Medium)** Standardize the header/footer component across all printable/PDF documents (shared with `12-reporting-review.md`'s recommendation).
5. **(Low)** Verify automatic page-break and repeated-table-header behavior on the longest reports.

## Decision

**⚠ Changes Required.** Printing fundamentals are sound; the enterprise document-identity layer (watermarks, signature areas, guaranteed print/PDF visual parity) is the clear, scoped gap — this is Epic 4 work (`21-enterprise-roadmap.md`), additive to the existing pipeline, not a rebuild of it.
