# Code Review Responsibilities

When reviewing implementation, the Claim Processing Expert must evaluate more than code correctness.

Every review must verify that the implementation preserves business integrity.

---

## Domain Review

Verify that:

Claim lifecycle matches the official state machine.

Aggregate boundaries are respected.

Business rules remain inside the domain layer.

No business logic exists inside controllers.

No business logic exists inside repositories.

Entities protect their own invariants.

Value Objects remain immutable.

State transitions are controlled.

Domain Events are emitted correctly.

---

## Application Review

Verify that:

Use Cases have a single responsibility.

Commands modify state only.

Queries never modify state.

Validation is deterministic.

Transactions are well defined.

External services are abstracted.

Workflow orchestration remains in the application layer.

No duplicated orchestration logic exists.

---

## Infrastructure Review

Verify that:

Repositories only persist aggregates.

Database logic does not replace domain logic.

External APIs are isolated.

Caching does not introduce inconsistency.

Retry policies are appropriate.

Logging is structured.

Tracing is available.

---

## API Review

Verify that APIs:

Are idempotent where required.

Return consistent error models.

Validate input before execution.

Do not expose internal implementation.

Support pagination.

Support filtering.

Support versioning.

Return proper HTTP status codes.

# Common Anti-Patterns

The Claim Processing Expert must reject implementations containing the following.

---

## Business Logic in Controllers

Incorrect

Controller calculates coverage.

Correct

Controller delegates to Application Layer.

---

## SQL Determines Business Rules

Incorrect

Stored procedures decide claim approval.

Correct

Domain Services decide approval.

---

## Mutable Historical Claims

Incorrect

UPDATE historical claim.

Correct

Create new version.

---

## Hardcoded Benefit Rules

Incorrect

if employer == "ABC"

Correct

Benefit Engine configuration.

---

## Direct Financial Calculation

Incorrect

Claim module calculates payable amounts independently.

Correct

Delegate calculations to Financial Engine and Benefit Engine.

---

## Hidden State Changes

Incorrect

Saving a claim silently changes benefit consumption.

Correct

Every change is explicit, transactional, and auditable.

---

## Physical Deletion

Incorrect

DELETE FROM Claims

Correct

Soft Delete only where permitted, otherwise immutable historical records.

---

## Bypassing Workflow

Incorrect

Update status directly in database.

Correct

Transition only through the Workflow Engine.

---
