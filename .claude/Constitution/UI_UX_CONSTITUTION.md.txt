# Waad TPA UI/UX Constitution

Version: 1.0

This document defines the mandatory user experience standards for the Waad TPA Platform.

The goal is not visual beauty.

The goal is operational excellence.

Every screen must improve productivity while preserving existing business workflows.

---

# Primary Design Goal

The platform is used by:

- Claim Officers
- Medical Reviewers
- Accountants
- Provider Staff
- Employer Administrators
- System Administrators

Users spend many hours every day inside the application.

Every design decision must reduce effort.

---

# Operational First

Always optimize for:

Less typing

Less clicking

Less navigation

Less waiting

Less mistakes

More visibility

More productivity

Beautiful interfaces without operational efficiency are considered poor design.

---

# Consistency

Every page must follow the same interaction model.

Users should never learn different behaviors for different modules.

Navigation

Buttons

Filters

Dialogs

Tables

Forms

Reports

Printing

must behave consistently.

---

# Layout

Default Layout

RTL

Language

Arabic

Responsive

Desktop First

Tablet Supported

Mobile only for simple operations.

Enterprise data-entry screens are optimized for desktop.

---

# Navigation

Navigation should always answer:

Where am I?

What can I do?

What happened?

What should I do next?

Avoid deep navigation.

Maximum recommended depth:

3 levels.

---

# Data Tables

Every business table should support:

Quick Search

Advanced Filters

Sorting

Pagination

Column Visibility

Column Resize

Column Reordering

Sticky Header

Row Selection

Bulk Actions

Export Excel

Export PDF

Print

Refresh

Totals (when applicable)

Loading Indicators

Empty State

Error State

Keyboard Navigation

Tables should remember:

Filters

Sorting

Visible Columns

Page Size

User Preferences

---

# Forms

Forms must be designed for fast data entry.

Every form should support:

Clear Sections

Keyboard Navigation

Tab Order

Validation

Inline Errors

Loading State

Save Progress (where applicable)

Cancel

Reset

Confirmation before destructive actions

Warn before leaving unsaved changes.

---

# Lookup Fields

Large datasets must never use simple dropdowns.

Use searchable lookup dialogs.

Examples:

Member

Provider

Employer

Benefit Policy

Medical Category

Medical Service

Provider Contract

Search first.

Select second.

---

# Validation

Validation should happen in layers.

Client

↓

API

↓

Domain

↓

Database

Never rely on a single validation layer.

Validation messages must explain:

What happened.

Why.

How to fix it.

---

# Dashboards

Dashboards should answer:

What needs attention?

What is overdue?

What requires approval?

What changed today?

Avoid decorative charts.

Prioritize actionable information.

---

# Search

Search is a core business feature.

Support:

Quick Search

Advanced Search

Saved Filters

Recent Searches

Search History

Partial Matching

Arabic Search

English Search

Barcode Search

Code Search

Searching should be fast.

---

# Reports

Reports are operational tools.

Not exports.

Every report should support:

Filters

Grouping

Totals

Subtotals

Drill Down

Export Excel

Export PDF

Print Preview

Report generation should be asynchronous for large datasets.

---

# Printing

Printing is a core business function.

Every printable document must support:

A4

RTL

Arabic Fonts

Professional Header

Professional Footer

Page Numbers

Print Date

Print Time

Company Logo

Signature Area

Stamp Area

Watermark Support

Draft Mode

Print Preview

Printed output must match PDF output.

---

# PDF

PDF documents are official business records.

Every generated PDF must support:

RTL

Unicode Arabic Fonts

Correct Arabic Text Shaping

Proper Line Wrapping

Automatic Page Breaks

Tables Across Pages

Professional Margins

Header

Footer

Page Numbers

Document Number

Generation Date

Generated By

Company Identity

PDF generation should produce archive-quality documents.

---

# Financial Display

Money should always display consistently.

Default Currency

Libyan Dinar (LYD)

Examples

1,250.500 د.ل

Negative values should be visually distinguishable.

Totals should always be emphasized.

---

# Dates

Display

DD/MM/YYYY

Time

24-hour format

Examples

08:30

16:45

Store internally using UTC where appropriate.

---

# Colors

Colors communicate status.

Green

Success

Blue

Information

Orange

Warning

Red

Error

Gray

Inactive

Never rely on color alone.

Always provide text or icons.

---

# Accessibility

Support:

Keyboard Navigation

Visible Focus

Readable Fonts

Sufficient Contrast

Clear Error Messages

Meaningful Icons

Accessibility improves productivity.

---

# Notifications

Notifications should be meaningful.

Always explain:

What happened

Why

Recommended action

Avoid technical messages.

---

# Loading Experience

Never leave users waiting without feedback.

Show:

Loading Indicators

Progress Bars

Estimated Progress (when possible)

Skeleton Loaders for large pages.

---

# Error Experience

Every error should contain:

Friendly Message

Technical Reference ID

Recovery Suggestion

Retry Option

Never expose stack traces.

---

# Attachments

Attachments should support:

Preview

Download

Replace

Version Awareness

File Size

Upload Progress

Supported Formats

---

# AI UI Behavior

Before modifying any screen Claude must ask:

Can this reduce clicks?

Can this reduce typing?

Can this improve readability?

Can this preserve workflow?

Can this improve printing?

Can this improve reporting?

Can this improve Arabic usability?

If not,

reconsider the implementation.

---

# Final Principle

The best interface is the one users stop thinking about.

The system should allow users to focus on claims, providers and members—not on understanding the software.
